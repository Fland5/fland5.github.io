性能优化

发表于 2020-04-24 | 更新于 2020-04-26 | 分类于 性能
  1. 代码
    1. 排除循环中利用“+”号进行字符串拼接工作，优先选用StringBuffer（线程安全）或者 StringBuilder（非线程安全）。
    2. 通过对StringBuffer与StringBuilder预设容量来避免二者在实际运行中频繁扩容从而导致的性能问题。
    3. 返回空值
      Collections.EMPTY_LIST;Collections.EMPTY_SET;Collections.EMPTY_MAP;Boolean.TRUE。
    4. 排除循环中操作数据库的工作
      代码循环，sql不循环。
      缓存思路。
    5. 避免循环体中的try catch.
    6. 锁优化
      减少锁持有时间
      （只在必要地方加锁，将锁从方法声明移到方法体中会延迟锁的加载，从而降低锁竞争的机率）
      锁分离
      读写锁ReadWriteLock,根据功能分离成读锁和写锁，这样读读不互斥，读写互斥，写写互斥，即保证了线程安全，又提高了性能。
    7. 缓存流的使用
    8. 减少前端请求次数
    9. 后端一次性可加载的数据，防止多次访问数据库。
  2. 数据库
    1. SQL语句全大写或者全小写。
    2. 子查询的执行效率不高，是因为执行子查询时数据库需要为内部嵌套的语句查询的结果建立一个临时表，然后再使用临时表中的数据进行查询。查询完成后再删除这个临时表。
      可以利用join语句替换子查询，提高效率，是因为join语句不需要建立临时表，所以其查询速度会优于子查询。
    3. 建立索引避免全表扫描（建议在where及order by涉及的列上建立索引）。
    4. 防止索引失效（避免在where子句中使用 != 或 <> 操作符）。
    5. 索引并不是越多越好，索引固然可以提高相应的 select 的效率，但同时也降低了 insert 和 update 的效率，因为 insert 或 update 时有可能会重建索引。
    6. 数字类型字段的性能优于字符类型，因此在设计只含数字的字段时不要使用字符类型。
    7. 不要使用 select from t ，而要选用具体的字段列表代替 ，不要返回用不到的任何字段。
    8. 应尽量避免在 where 子句中使用 or 来连接条件,可以考虑使用 union 代替。
    9. in 和 not in 也要慎用。对于连续的数值，能用 between 就不要用in，可用exists 代替 in。
    10. 尽量避免在 where 子句中对字段进行表达式操作或函数操作。
    11. 在Join表的时候字段使用相同类型，并将其索引。（INT与DECIMAL字段join就无法使用该字段索引，STRING类型要求字符集一致才行）
  3. 服务器
    1. top命令。
    2. w命令。
    3. iostat命令。
    4. free m命令。
    5. mpstat命令。
